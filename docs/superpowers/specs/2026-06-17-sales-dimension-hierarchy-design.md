# Sales Dimension Hierarchy Design

日期：2026-06-17

## 1. 背景

当前销售业绩分析 demo 已支持“主维度汇总列表 + 抽屉多维拆解”，但维度组合规则过于粗糙：系统只按维度组判断合法性，同组维度一律禁止互相拆解。

这个规则会误伤合理场景。例如“部门”作为主维度时，用户希望继续下钻查看部门内各咨询师的业绩，这是正常的经营分析路径；但“咨询师”作为主维度时，再反向查看部门汇总则不符合直觉。

本次设计目标是把维度规则从“同组互斥”升级为“跨组默认允许 + 同组按上级到下级下钻”，并新增日期、项目、项目分类三个维度。

## 2. 目标

- 新增“日期、项目分类、项目”三个分析维度。
- 支持上级维度下钻到下级维度。
- 禁止下级维度反向下钻到上级维度。
- 保留跨组维度默认互相拆解的能力。
- 继续使用当前主表、抽屉和 Tab 交互，不引入新的报表形态。

## 3. 不做范围

- 不新增图表、看板配置器或自定义报表。
- 不新增周、月等日期粒度切换。
- 不改变 21 个指标口径。
- 不接入真实 CRM 接口。
- 不新增权限或保存视图能力。

## 4. 维度模型

维度从 6 个扩展为 9 个：

| 维度组 | 维度 | 层级关系 | 说明 |
| --- | --- | --- | --- |
| 组织 | 部门、咨询师 | 部门 -> 咨询师 | 部门是组织汇总，咨询师是个人明细 |
| 来源 | 渠道分类、渠道 | 渠道分类 -> 渠道 | 渠道分类是来源大类，渠道是具体来源 |
| 地域/履约 | 城市、机构 | 城市 -> 机构 | 城市是区域汇总，机构是具体服务单元 |
| 项目 | 项目分类、项目 | 项目分类 -> 项目 | 项目分类是业务大类，项目是具体产品或服务 |
| 时间 | 日期 | 无层级 | 日期按日聚合展示 |

维度配置继续保留 `group`，并增加可表达层级顺序的字段，例如：

```ts
{
  key: 'consultant',
  label: '咨询师',
  group: 'org',
  level: 2,
}
```

同组内 `level` 数字越小，表示维度越上层；数字越大，表示维度越明细。日期维度不需要上下级关系。

## 5. 合法下钻规则

合法关系由规则自动推导，不手写完整组合矩阵。

规则如下：

- 自身维度不能下钻自身。
- 跨组维度默认允许互相拆解。
- 同组维度只允许上级下钻下级。
- 同组维度禁止下级反向下钻上级。
- 同组同级维度禁止互相下钻。
- 日期可以与所有非日期业务维度互相拆解，但日期不能拆日期。

关键组合如下：

| 主维度 | 允许 | 禁止 |
| --- | --- | --- |
| 部门 | 咨询师 | 部门 |
| 咨询师 | 跨组维度、日期 | 部门、咨询师 |
| 渠道分类 | 渠道 | 渠道分类 |
| 渠道 | 跨组维度、日期 | 渠道分类、渠道 |
| 城市 | 机构 | 城市 |
| 机构 | 跨组维度、日期 | 城市、机构 |
| 项目分类 | 项目 | 项目分类 |
| 项目 | 跨组维度、日期 | 项目分类、项目 |
| 日期 | 所有非日期业务维度 | 日期 |

## 6. 数据字段

当前事实数据已包含 `dealDate`，可以直接作为“日期”维度值。日期维度 v1 按日聚合，例如 `2026-06-17`。

项目相关需要在事实数据中新增两个字段：

```ts
projectCategory: string;
project: string;
```

示例：

```ts
{
  projectCategory: '高端咨询',
  project: '年度管理咨询包',
}
```

## 7. 交互变化

主维度下拉从 6 个选项扩展为 9 个选项：

- 日期
- 部门
- 咨询师
- 渠道分类
- 渠道
- 城市
- 机构
- 项目分类
- 项目

用户选择主维度后，主表仍按当前主维度聚合展示。点击“查看拆解”后，抽屉 Tab 根据新下钻规则自动生成。

示例：

- 主维度为“部门”时，抽屉展示“咨询师”以及其他跨组维度。
- 主维度为“咨询师”时，抽屉不展示“部门”。
- 主维度为“项目分类”时，抽屉展示“项目”以及其他跨组维度。
- 主维度为“项目”时，抽屉不展示“项目分类”。
- 主维度为“日期”时，抽屉展示所有非日期业务维度。

顶部“统计时间”筛选继续表示统计范围；“日期”维度表示在这个范围内按日查看业绩分布。二者语义不同，可以同时存在。

## 8. 测试口径

需要覆盖以下验证：

- 维度列表包含 9 个主维度。
- 部门可以下钻咨询师，咨询师不能下钻部门。
- 渠道分类可以下钻渠道，渠道不能下钻渠道分类。
- 城市可以下钻机构，机构不能下钻城市。
- 项目分类可以下钻项目，项目不能下钻项目分类。
- 日期可以与业务维度互相拆解，但不能下钻日期自身。
- 主表聚合可以按日期、项目分类、项目运行。
- 抽屉拆解可以按新增维度运行。

## 9. 成功标准

- 用户可以选择日期、项目分类、项目作为主维度。
- 抽屉 Tab 能体现“上级到下级”的有向下钻规则。
- 原有跨组拆解能力不丢失。
- 不再使用“同组维度一律禁止”的旧规则。
- 新增维度不会改变既有指标计算口径。
