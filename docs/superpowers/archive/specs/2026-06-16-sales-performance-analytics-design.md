# CRM 业绩统计模块重构设计

日期：2026-06-16

## 1. 背景与问题

当前 CRM 业绩统计模块以“咨询师业绩”为固定主视角，业务方一旦提出类似“某咨询师在某机构的业绩产出”“某咨询师在某渠道的业绩产出”等问题，现有看板无法直接支持，只能由产研团队不断补充固定看板或临时统计。

这个问题的本质不是缺少某几个看板，而是统计模块仍停留在“固定报表”形态，没有把业绩数据抽象成可按维度拆解的分析模型。继续补页面会让看板越来越碎，也会让产研团队持续陷入补丁式开发。

本次重构目标是把业绩统计模块升级为“主维度汇总列表 + 抽屉多维拆解”的受控分析工具。业务可以先选择一个主维度查看整体表现，再进入某个对象查看它在其他合法维度下的业绩拆解。

## 2. 第一版目标

第一版聚焦解决“维度组合灵活”问题，不做完整 CRM 看板闭环。

目标：

- 支持按单个主维度查看业绩汇总。
- 支持在某个主维度对象下，按合法的其他维度查看拆解数据。
- 指标固定展示一组业绩、成交、新诊、复购、新客转化和历史复购贡献指标。
- 基于 Ant Design 5.0 的常规组件设计，避免 demo 交互在正式系统中无法落地。
- 通过 demo 表达产品形态和交互方案，不接入真实 CRM 接口。

不做：

- 不做在线 Excel 或自由拖拽透视表。
- 不做三维及以上组合。
- 不做指标自定义配置。
- 不做图表联动。
- 不做看板保存、权限体系、真实接口和完整 CRM 集成。

## 3. 维度模型

第一版包含 6 个维度：

| 维度组 | 维度 | 说明 |
| --- | --- | --- |
| 人员组织侧 | 咨询师、部门 | 谁产生或承接了业绩 |
| 来源侧 | 渠道、渠道分类 | 业绩从哪里来 |
| 履约/区域侧 | 机构、城市 | 业绩最终落在哪个服务或地域单元 |

合法组合规则：

- 允许跨维度组拆解。
- 不允许同组维度互相拆解。

因此合法关系为：

| 主维度 | 可拆解维度 |
| --- | --- |
| 咨询师 | 渠道、渠道分类、机构、城市 |
| 部门 | 渠道、渠道分类、机构、城市 |
| 渠道 | 咨询师、部门、机构、城市 |
| 渠道分类 | 咨询师、部门、机构、城市 |
| 机构 | 咨询师、部门、渠道、渠道分类 |
| 城市 | 咨询师、部门、渠道、渠道分类 |

这等价于配置规则：

```ts
primaryDimension.group !== breakdownDimension.group
```

这种设计比手写所有组合更容易扩展。未来新增维度时，只要明确它属于哪个维度组，就可以自动推导可拆解维度。

## 4. 指标口径

第一版固定展示 21 个指标。主列表和抽屉拆解表使用同一套指标，避免用户在汇总视图和拆解视图之间切换时看到不同口径。

指标按业务含义分组展示：

| 指标组 | 指标 |
| --- | --- |
| 基础业绩 | 上报业绩、确认业绩 |
| 成交基础 | 成交单量、成交客户数、客单价 |
| 新诊 | 新诊业绩、新诊成交单量、新诊成交客户数、新诊成交单量占比、新诊成交客户占比、新诊成交业绩占比 |
| 复购 | 复购业绩、复购成交单量、复购成交客户数、复购成交单量占比、复购成交客户占比、复购成交业绩占比 |
| 新客转化 | 新客成交率、新客业绩贡献占比 |
| 历史复购贡献 | 历史复购客户当期贡献率、历史复购业绩当期贡献率 |

demo 口径：

| 指标 | demo 计算口径 |
| --- | --- |
| 上报业绩 | 统计范围内上报金额求和 |
| 确认业绩 | 统计范围内确认金额求和，仅独立展示，不参与业绩占比类指标计算 |
| 成交单量 | 统计范围内成交记录数 |
| 成交客户数 | 统计范围内成交客户去重数 |
| 客单价 | 上报业绩 / 成交单量 |
| 新诊业绩 | 成交类型为新诊的上报业绩求和 |
| 新诊成交单量 | 成交类型为新诊的成交记录数 |
| 新诊成交客户数 | 成交类型为新诊的成交客户去重数 |
| 复购业绩 | 成交类型为复购的上报业绩求和 |
| 复购成交单量 | 成交类型为复购的成交记录数 |
| 复购成交客户数 | 成交类型为复购的成交客户去重数 |
| 新诊成交单量占比 | 新诊成交单量 / 成交单量 |
| 复购成交单量占比 | 复购成交单量 / 成交单量 |
| 新诊成交客户占比 | 新诊成交客户数 / 成交客户数 |
| 复购成交客户占比 | 复购成交客户数 / 成交客户数 |
| 新诊成交业绩占比 | 新诊业绩 / 上报业绩 |
| 复购成交业绩占比 | 复购业绩 / 上报业绩 |
| 新客成交率 | 选定时间内创建且成交的客户数 / 选定时间内创建的客户数 |
| 新客业绩贡献占比 | 选定时间内创建且成交客户的上报业绩 / 上报业绩 |
| 历史复购客户当期贡献率 | 当期复购客户数 / 历史总复购客户数 |
| 历史复购业绩当期贡献率 | 当期复购业绩 / 历史总复购业绩 |

所有带“业绩、金额、贡献”的统计指标均按上报业绩计算。确认业绩只作为独立展示字段，用于对比上报口径和确认口径。

## 5. 页面结构

页面采用 Ant Design 5.0 可落地组件设计。

### 5.1 筛选区

组件建议：

- `Form`
- `DatePicker.RangePicker`
- `Select`
- `Button`

作用：

- 限定统计范围。
- 例如时间、业绩状态、业务范围等。
- 筛选区不参与维度组合，只决定统计哪批业绩数据。

### 5.2 主维度选择

组件建议：

- `Select`

可选主维度：

- 咨询师
- 部门
- 渠道
- 渠道分类
- 机构
- 城市

用户选择主维度后，主列表按该维度聚合展示。

### 5.3 主列表

组件建议：

- `Table`

表格列：

| 列 | 说明 |
| --- | --- |
| 主维度名称 | 随主维度变化，例如咨询师、部门、渠道 |
| 基础业绩 | 当前主维度对象的上报业绩、确认业绩 |
| 成交基础 | 当前主维度对象的成交单量、成交客户数、客单价 |
| 新诊 | 当前主维度对象的新诊业绩、单量、客户数和占比 |
| 复购 | 当前主维度对象的复购业绩、单量、客户数和占比 |
| 新客转化 | 当前主维度对象的新客成交率、新客业绩贡献占比 |
| 历史复购贡献 | 当前主维度对象的历史复购客户和业绩当期贡献率 |
| 操作 | 查看拆解 |

示例：

| 咨询师 | 上报业绩 | 确认业绩 | 成交单量 | 成交客户数 | 客单价 | ... | 操作 |
| --- | ---: | ---: | ---: | ---: | ---: | --- | --- |
| 张敏 | 158.0万 | 143.9万 | 5 | 4 | 31.6万 | ... | 查看拆解 |
| 李然 | 146.0万 | 133.2万 | 4 | 4 | 36.5万 | ... | 查看拆解 |

主列表职责是回答：“当前主维度下，各对象整体表现如何？”

### 5.4 拆解抽屉

组件建议：

- `Drawer`
- `Tabs`
- `Table`

点击主列表某一行的“查看拆解”后，打开右侧抽屉。抽屉标题展示当前对象，例如：

```text
张敏 · 业绩拆解
```

抽屉内使用 `Tabs` 展示当前主维度允许拆解的其他维度。

示例：主维度为“咨询师”时，Tab 为：

- 渠道
- 渠道分类
- 机构
- 城市

主维度为“渠道”时，Tab 为：

- 咨询师
- 部门
- 机构
- 城市

每个 Tab 内是一张拆解表格。

拆解表格列：

| 列 | 说明 |
| --- | --- |
| 拆解维度名称 | 随 Tab 变化，例如渠道、机构、城市 |
| 基础业绩 | 当前拆解维度对象的上报业绩、确认业绩 |
| 成交基础 | 当前拆解维度对象的成交单量、成交客户数、客单价 |
| 新诊 | 当前拆解维度对象的新诊业绩、单量、客户数和占比 |
| 复购 | 当前拆解维度对象的复购业绩、单量、客户数和占比 |
| 新客转化 | 当前拆解维度对象的新客成交率、新客业绩贡献占比 |
| 历史复购贡献 | 当前拆解维度对象的历史复购客户和业绩当期贡献率 |

示例：张敏按渠道拆解：

| 渠道 | 上报业绩 | 确认业绩 | 成交单量 | 成交客户数 | 客单价 | ... |
| --- | ---: | ---: | ---: | ---: | ---: | --- |
| 信息流 | 90.0万 | 82.4万 | 2 | 2 | 45.0万 | ... |
| 私域 | 35.0万 | 31.8万 | 1 | 1 | 35.0万 | ... |
| 转介绍 | 20.0万 | 18.6万 | 1 | 1 | 20.0万 | ... |

## 6. 关键交互

- 用户先选择主维度。
- 系统按主维度展示汇总列表。
- 用户点击某一行“查看拆解”。
- 系统打开抽屉，并根据主维度自动生成合法拆解 Tab。
- 用户在抽屉内切换 Tab，查看该对象在不同维度下的拆解数据。
- 主列表支持导出当前汇总视图。
- 抽屉内支持导出当前 Tab 的拆解数据。

非法组合不提供入口。例如主维度是“咨询师”时，不展示“部门”拆解 Tab。

## 7. 数据模型

业绩事实数据可以理解为一行一条业绩记录，至少包含：

| 字段 | 示例 |
| --- | --- |
| 咨询师 | 张敏 |
| 部门 | 华东一部 |
| 渠道 | 信息流 |
| 渠道分类 | 线上投放 |
| 机构 | 上海中心 |
| 城市 | 上海 |
| 客户ID | C001 |
| 上报金额 | 35000 |
| 确认金额 | 32800 |
| 成交类型 | 新诊 |
| 是否选定时间内创建客户 | 是 |
| 历史总复购客户数 | 120 |
| 历史总复购业绩 | 3600000 |
| 成交时间 | 2026-06-10 |
| 业绩状态 | 已成交 |

维度配置示例：

```ts
[
  { key: 'consultant', label: '咨询师', group: 'org' },
  { key: 'department', label: '部门', group: 'org' },
  { key: 'channel', label: '渠道', group: 'source' },
  { key: 'channelCategory', label: '渠道分类', group: 'source' },
  { key: 'institution', label: '机构', group: 'location' },
  { key: 'city', label: '城市', group: 'location' }
]
```

指标配置示例：

```ts
[
  { key: 'reportedAmount', label: '上报业绩', group: '基础业绩', aggregate: 'sum(reportedAmount)' },
  { key: 'confirmedAmount', label: '确认业绩', group: '基础业绩', aggregate: 'sum(confirmedAmount)' },
  { key: 'dealCount', label: '成交单量', group: '成交基础', aggregate: 'count(dealId)' },
  { key: 'customerCount', label: '成交客户数', group: '成交基础', aggregate: 'countDistinct(customerId)' },
  { key: 'averageDealAmount', label: '客单价', group: '成交基础', formula: 'reportedAmount / dealCount' }
]
```

## 8. 查询结构

主列表查询：

```ts
{
  primaryDimension: 'consultant',
  filters: {
    dateRange: ['2026-06-01', '2026-06-30'],
    dealStatus: 'closed'
  }
}
```

返回主维度汇总：

```ts
[
  {
    key: 'consultant:zhangmin',
    primaryDimensionValue: '张敏',
    reportedAmount: 1580000,
    confirmedAmount: 1439000,
    dealCount: 5,
    customerCount: 4
  }
]
```

抽屉拆解查询：

```ts
{
  primaryDimension: 'consultant',
  primaryDimensionValue: 'zhangmin',
  breakdownDimension: 'channel',
  filters: {
    dateRange: ['2026-06-01', '2026-06-30'],
    dealStatus: 'closed'
  }
}
```

返回拆解数据：

```ts
[
  {
    key: 'channel:info',
    breakdownDimensionValue: '信息流',
    reportedAmount: 900000,
    confirmedAmount: 824000,
    dealCount: 2,
    customerCount: 2
  }
]
```

## 9. 推荐 demo 范围

demo 用于表达交互方案，不需要接入真实 CRM。

需要呈现：

- 筛选区。
- 主维度选择。
- 主列表汇总。
- 查看拆解按钮。
- 抽屉。
- 抽屉内 Tab 切换拆解维度。
- 拆解表格。
- 分组表头指标宽表。
- 导出入口位置。

不需要实现：

- 真实接口。
- 真实导出文件。
- 权限控制。
- 看板保存。
- 图表。
- 自定义指标。
- 三维组合。
- 自由拖拽透视表。

## 10. 方案取舍

本设计放弃“主表直接二维组合”的树形表格方案，原因是：

- 主表直接二维组合会让页面一开始就承载两个维度，列表较重。
- 后续新增更多拆解维度时，主表的选择和展示复杂度会增加。
- “主维度汇总列表 + 抽屉多维拆解”更符合业务先看整体、再看某个对象拆解的阅读路径。

本设计也不采用“展开行内嵌子表”作为主方案，原因是：

- 多行展开后页面会变长。
- 全局排序、分页、导出更复杂。
- 抽屉能把拆解信息从主列表中分离出来，让主列表保持轻量。

最终推荐：

```text
Ant Design 5.0 Table 主列表
+ Drawer 抽屉
+ Tabs 维度切换
+ Table 拆解明细
```

## 11. 成功标准

当 demo 完成后，应能让产研和业务明确理解：

- 业绩统计模块不再靠新增固定看板响应每个维度诉求。
- 主列表可以切换不同主维度查看汇总。
- 单个对象可以在抽屉中按合法维度继续拆解。
- 维度组合受到规则控制，不是任意组合。
- 第一版指标固定为 21 个字段，并按分组表头展示。
- 该交互可以用 Ant Design 5.0 常规组件实现。
